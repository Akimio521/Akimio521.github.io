推一个自己的小项目——AutoFilm

发表于2023-10-02|更新于2023-10-02|文档
|字数总计:1.7k|阅读时长:5分钟|阅读量:|评论数:

前言

不知道大家家里也没有自己的NAS，我是今年年初的时候入了坑，学习了很多NAS的一些使用方式，感觉我的NAS不仅仅是一个网络硬盘，更多的是一个服务器，上面搭载了很多服务，像是密码管理器—Bitwarden、私人云盘——Nextcloud、网盘集合——Alist以及更为常见的影音媒体库Jellyfin

AutoFilm

如果大家和我一样属仓鼠的话那么一定很喜欢到处收集电影动漫什么的吧，但是奈何家境贫寒硬盘太小，放不下那么多自己喜欢的电影动漫，但是偶然间发现Emby和Jellyfin支持对远程视频链接的处理，于是自己尝试做了这个小项目

Github

正好最近改名了，Github从 zui-lang 改到了 Akimio521 大家不要走错了
AutoFilm-Github仓库地址

开发历程

我的NAS上搭建了Alist和Jellyfin两个服务器，我的音乐库在阿里云盘上，最开始是利用Alist挂载阿里云盘然后Rclone到本地挂载到Jellyfin上，但是每次Jellyfin扫库时，NAS的运行压力就会变大，而且观看体验并不好

在浏览博客间偶然发现Jellyfin可以利用 strm 文件代替视频文件，于是有了这个仓库，Emby会访问 strm 文件，其中有视频源文件的播放地址，Jellyfin服务器就会把这个地址发给客户端让客户端让客户端自行尝试播放

整个python程序的结构很简单，打开对应的Webdav服务器，遇到MP4,MKV,FLV,AVI视频文件就在对应本地文件夹输出一个同名strm文件，其内容为视频的链接，其中Alist需要关闭签名，关闭签名后，直链需要把URL中的/dav/改为/d/；如果遇到ASS,SRT,SSA字幕文件或JPG刮削图像或NFO视频信息文件就直接下载到对应文件夹中，方便Emby的视频读取

至此，整个AutoFilm输出文件夹就可以代替原本的Jellyfin视频媒体库了，如果Alist开启了302重定向，Emby客户端会跳转访问阿里云盘的地址，因此访问速度与Alist和Jellyfin服务器完全不相干，如果NAS是采用CloudFlare进行内网穿透的话，播放效果也不再受到影响

不同影音服务器的不同处理

  • Emby：服务器下载直链视频，并提交客户端进行流控，连接速度同时受限于Emby服务器的上下行带宽
  • Jellyfin：直接提交给客户端播放器，连接速度不受Jellyfin服务器带宽限制
  • Plex：无法识别，处理不了

Emby官方文档——Emby对Strm的处理
可见Jellyfin是最适合这种玩法的，客户端可以直接访问视频源地址，相当于Jellyfin帮助了客户端和视频进行 P2P打洞
Emby的效果可能会略逊一筹，但好在Emby是服务器获取视频资源，可以对视频进行转码后再推流给客户端，如果说你的视频资源在OneDrive，而影音服务器又搭建在国外的VPS上，使用Emby这种方式就可以让服务器代理访问资源，体验应该比直连OneDrive好一点（前提是VPS的网络要好）

必要参数

  • webdav_url：webdav服务器地址
  • username：webdav账号
  • password：webdav密码

非必要参数

  • output_path：输出文件目录，默认为当前文件夹的子目录Media
  • subtitle：是否下载字幕文件，默认true
  • nfo：是否下载NFO文件，默认false
  • img：是否下载JPG和PNG图片文件，默认 false

关于大小

用七米蓝的动漫库实测，25T的动漫库，默认配置只会占用1G的大小，相当于只需要1G的空间就能放下25T的视频库

关于Github Action

本来一开始是想让Github Action帮我完成所有资源的下载，但我Python写的是真的烂，再加上七米蓝大佬的仓库实在是太大，目前我发现只有纪录片库是可以完整下载的，其他的仓库由于文件太多，运行时间往往会超过6小时还没完成，Github就会ban掉我的Action，这里也是期待一个小伙伴帮我修改一下我的程序，提高一点运行效率。所以目前大家还是把程序下载下来放到本地或者是VPS上使用吧

其他

如大家所见，这个程序对已经规范命名、整理的视频盘的处理比较好，对于所有文件扔一堆的视盘，想刮削的话只能任凭Emby、Jellyfin本事了

解决方法

AutoFilm更新

这个只能说随缘了，有时间我研究一下怎么将视频文件名发给TMDB并规范重命名并刮削视频后保存到本地，但这个计划只能说是遥遥无期，也欢迎有能力的小伙伴提交你的PR。并且我估计如果使用这种方式估计Github Action就可以完全抛弃了（虽然可以设置一个参数是否要去TMDB刮削）

曲线救国

可以先找一个大硬盘的机子，可以是主力电脑，先将网盘上的资源下载一部分下来，然后扔给有刮削整理媒体库的程序（例如NasTools）然后再将整理好的文件夹上传回网盘（如果网盘有哈希的话应该会挺快的）最后再利用AutoFilm生成已经完美刮削的Strm媒体库，然后直接给Jellyfin或者Emby就行
